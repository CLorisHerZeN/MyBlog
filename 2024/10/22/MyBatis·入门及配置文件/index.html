概述

  • 是一个基于Java的持久层（对数据库进行操作）框架
  • JDBC步骤：编写sql ==》 预编译 ==》 设置参数 ==》 执行sql ==》 封装结果
  • 工具（之前的JDBCTemplate等）：
    • 功能简单，sql语句编写在代码内，硬编码高耦合方式
  • MyBatis框架：整体解决方案，半自动框架
    • sql语句支持定制，写在配置文件中，与Java编码分离
    • 剩下步骤由其自动完成

小结

  • 原生：Dao ==》 DaoImpl

    myBatis：Mapper ==》 xxMapper.xml

  • SqlSession代表和数据库的一次对话，用完必须关闭

  • SqlSession与connection一样，都是非线程安全，因此不能直接提取为成员变量共享，每次使用都应获取新的对象

  • mapper接口没有实现类，但将接口与xml进行绑定后，myBatis会为接口自动创建代理对象

  • 两个重要配置文件：

    • mybatis的全局配置文件：包含数据库连接池信息，事务管理器信息等运行环境信息
    • sql映射文件：必要文件，保存了每一个sql语句的映射信息，mybatis通过改文件把sql抽取出来

#核心配置文件详解

全局配置文件

引入dtd约束

  • 在该文件中引入了约束文件，规定xml标签语法规则（提示信息）：http://mybatis.org/dtd/mybatis-3-config.dtd

核心配置文件中的标签必须按照固定的顺序：

  • configuration（配置）

    • properties（属性）：引入外部properties配置文件的内容（了解，少用）

      • resource属性：引入类路径下资源
      • url属性：引入网络路径或者磁盘路径下资源

    • settings（设置）：改变MyBatis运行时行为，setting用于设置每一个设置项

      • 设置项参考：配置_MyBatis中文网
      • 举例mapUnderscoreToCamelCase：

    • typeAliases（类型别名）：为Java类型起别名，别名不区分大小写，不要占用内部已起的别名

    • typeHandlers（类型处理器）

    • objectFactory（对象工厂）

    • plugins（插件）

    • environments（环境配置）：配置多种环境

      • environment（环境变量）：配置具体环境信息，必须有以下两个标签
        • transactionManager（事务管理器）
        • dataSource（数据源）

    • databaseIdProvider（数据库厂商标识）：支持多数据库厂商，可以根据标识执行不同sql

      在映射文件中标识该sql语句的使用环境：databaseId="mysql"

    • mappers（映射器）：将sql映射注册到全局配置中

      注册配置文件：

      • resource：引用类路径下的sql映射文件
      • url：引用网络路径/磁盘路径下的sql映射文件

      注册接口：

      • class：引用/注册接口
        1. 有sql映射文件，映射文件名必须和接口同名，且放在接口同一目录下
        2. 无sql映射文件，所有sql都是利用注解写在接口上（不推荐，比较重要的sql还是写映射文件）

映射文件

映射文件指导MyBatis如何进行数据库增删改查

SQL 映射文件只有很少的几个顶级元素（按照应被定义的顺序列出）：

  • cache – 该命名空间的缓存配置。
  • cache-ref – 引用其它命名空间的缓存配置。
  • resultMap – 描述如何从数据库结果集中加载对象，是最复杂也是最强大的元素。
  • sql – 可被其它语句引用的可重用语句块。
  • insert – 映射插入语句。
  • update – 映射更新语句。
  • delete – 映射删除语句。
  • select – 映射查询语句。

参数处理

  1. 单个参数：#{参数名}，取出参数值，mybatis不会特殊处理

    可以使用${}和#{}以名称获取参数的值

  2. 多个参数：此时MyBatis会自动将这些参数放在一个map集合中，param1,param2…为键，以参数为值

    • key：param1…paramN，或者写参数的索引

      sql语句内写：#{param1}，访问map集合的键就可以获取相对应的值

    • 推荐命名参数：明确封装参数时map的key@Param("id")（写在接口方法的参数前）；

      key：使用@Param指定的值，value：参数值

      #{指定的key}取出对应的参数值

    • 也可直接传入map，只需要通过${}和#{}访问map集合的键就可以获取相对应的值

  3. #{}：以预编译的形式将参数设置到sql语句中

    ${}：将取出的值直接拼装在sql语句中，会有安全问题

    大多数情况下都有#{}，若分表时设置表名，原生jdbc不支持占位符的地方就可以使用${}取值

select

  1. 查询一个实体类对象

  2. 查询一个list集合：返回值类型写集合中元素的类型

  3. 查询返回一条记录的map：key为列名，value为对应值，结果类型直接写map

    查询返回多条记录的map

    • 将表中的数据以map集合的方式查询，一条数据对应一个map；若有多条数据，就会产生多个map集合，并且最终要以一个map的方式返回数据，此时需要通过@MapKey注解设置map集合的键，值是每条数据所对应的map集合

resultMap属性：

自定义结果集映射规则，不能与resultType一起用

  1. 处理字段和属性的映射关系

    若字段名和实体类中的属性名不一致，则可以通过resultMap设置自定义映射

  2. 多对一映射处理

    情景：查询员工信息以及员工所对应的部门信息

    • 级联方式处理映射关系
    • 使用association处理映射关系：可以指定联合的JavaBean对象
    • 使用association进行分步查询
    1. 查询员工信息
    1. 根据员工所对应的部门id查询部门信息

分步查询的优点：可以实现延迟加载，但是必须在核心配置文件中设置全局配置信息：

  • lazyLoadingEnabled：延迟加载的全局开关。当开启时，所有关联对象都会延迟加载

  • aggressiveLazyLoading：当开启时，任何方法的调用都会加载该对象的所有属性。 否则，每个属性会按需加载

    此时就可以实现按需加载，获取的数据是什么，就只会执行相应的sql。此时可通过association和

  • collection中的fetchType属性设置当前的分步查询是否使用延迟加载，fetchType=”lazy(延迟加
    载)|eager(立即加载)”
