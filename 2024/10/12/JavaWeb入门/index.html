XML

  • EXtensible Markup Language，可扩展标记语言，与html一样都是标记语言，基本语法都是标签

作用：

  1. 用来保存数据，且保存的数据具有自我描述性；
  2. 作为项目或模块的配置文件；
  3. 作为网络传输数据的格式（现在以JSON为主）

常见配置文件类型：

  • Properties文件：这是一种非常常见的配置文件格式，主要在Java应用程序中使用。它以简单的键值对形式存储数据，通常具有.properties扩展名。但在表达复杂的多层的配置信息时较为吃力
  • XML文件：是一种用于存储和传输数据的标记语言。它被许多不同的应用程序和框架用于配置，因为它具有良好的可读性和结构化。
  • JSON文件：（JavaScript Object Notation）是一种轻量级的数据交换格式，易于人阅读和编写，同时也易于机器解析和生成。在现代Web服务和应用程序中非常流行。

XML基本语法

特点

  • 文件中的标签通常是自定义的，可以根据自己的需求定义标签（也称为元素）和文档结构。XML不预定义任何特定的标签集。

  • XML 中的每个元素都是成对出现的，有开始和结束，自闭和标签除外，但是都得有 ‘/‘结束标志

    如：<student>xxxxx</student> 自闭和：<student name='xxx' />

  • 每个XML文档都有且只有一个根元素（Root Element）

  • XML标签对大小写敏感

  • XML必须正确嵌套

  • 同级标签以所缩进对齐

标签命名规则

  1. 名称可以包含字母、数字以及其他的字符。

  2. 名称不能以数字或者标点符号开始。

  3. 名称不能以字母 xml（或者 XML、Xml 等等）开始。

  4. 名称不能包含空格，或者 :号

  5. 可使用任何名称，没有保留的字词。

  6. 如出现特殊字符需要转义 如：<、>、”、’、&等

XML文档声明

  • 可选，但文档声明一定要从第一行第一列开始写

XML元素

  • 根标签

    • 根标签有且只能有一个。

  • 标签关闭

    • 双标签：开始标签和结束标签必须成对出现。
    • 单标签：单标签在标签内关闭。

  • 标签嵌套

    • 可以嵌套，但是不能交叉嵌套。

      如：<age>23<name>zhangsan></age></name>是错误的

XML属性

  • 标签名、属性名建议使用小写字母
  • 属性
    • 书写在标签内，对标签的数据进行扩展。对标签的进一步描述。
    • <标签名 属性名=“属性值” 属性名=“属性值”> </标签名> 属性名也是自定义的
    • 如果是双标签，属性要书写在开始标签内
    • 属性名不要出现空格，；或 ： 特殊字符不要出现
    • 属性必须有值
    • 属性值必须加引号，单或双

XML注释

  • 与HTML一致<!-- -->
  • 注释不能嵌套
  • 书写注释的时候，内容尽可能的去避免 -- 字符

文本区域（CDATA区）

  • 向XML解析器表示其内部文本内容只是纯文本，无需XML语言解析
  • 格式：<![CDATA[文本内容]]>

XML约束

  • 我们需要根据XML约束中的规定来编写XML配置文件，idea会在程序员编写XML的时候根据约束来提示
  • DTD：语法简单，约束没那么细致
  • Schema：上手慢，约束细致。程序员无需亲自写约束，而是写配置内容

HTTP协议

  • HTTP：Hyper Text Transfer Protocol(超文本传输协议)，规定了浏览器与服务器之间数据传输的规则。
  • 协议中的数据又叫报文

HTTP协议特点：

  • **基于TCP协议: ** 面向连接，安全

    TCP是一种面向连接的（建立连接之前是需要经过三次握手）、可靠的、基于字节流的传输层通信协议，在数据传输方面更安全

  • 基于请求-响应模型: 一次请求对应一次响应（先请求后响应）

  • HTTP协议是无状态协议: 对于数据没有记忆能力。每次请求-响应都是独立的

    无状态指的是客户端发送HTTP请求给服务端之后，服务端根据请求响应数据，响应完后，不会记录任何信息。

    • 缺点: 多次请求间不能共享数据
    • 优点: 速度快

浏览器和服务器是按照HTTP协议进行数据通信的。

HTTP协议又分为：请求协议和响应协议

客户端发送给服务器的称为”请求报文**”，**服务器发送给客户端的称为”响应报文”

  • 请求协议：浏览器将数据以请求格式发送到服务器
    • 包括：请求行、请求头 、请求体
  • 响应协议：服务器将数据以响应格式返回给浏览器
    • 包括：响应行 、响应头 、响应体

请求协议

请求行

  • 常见请求方式：常用的是 ：GET、POST
请求方式 请求说明
GET 获取资源。
向特定的资源发出请求。
POST 传输实体主体。
向指定资源提交数据进行处理请求（例：上传文件），数据被包含在请求体中。
OPTIONS 返回服务器针对特定资源所支持的HTTP请求方式。
因为并不是所有的服务器都支持规定的方法，为了安全有些服务器可能会禁止掉一些方法，例如：DELETE、PUT等。那么OPTIONS就是用来询问服务器支持的方法。
HEAD 获得报文首部。
HEAD方法类似GET方法，但是不同的是HEAD方法不要求返回数据。通常用于确认URI的有效性及资源更新时间等。
PUT 传输文件。
PUT方法用来传输文件。类似FTP协议，文件内容包含在请求报文的实体中，然后请求保存到URL指定的服务器位置。
DELETE 删除文件。
请求服务器删除Request-URI所标识的资源
TRACE 追踪路径。
回显服务器收到的请求，主要用于测试或诊断
CONNECT 要求用隧道协议连接代理。
HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器

  • GET请求和POST请求的区别：

    区别方式 GET请求 POST请求
    请求参数 请求参数在请求行中。
    例：/brand/findAll?name=OPPO&status=1    		 请求参数在请求体中
    请求参数长度 请求参数长度有限制(浏览器不同限制也不同) 请求参数长度没有限制
    安全性 安全性低。原因：请求参数暴露在浏览器地址栏中。 安全性相对高

  • 用途区别

    • GET请求：通常用于请求服务器发送资源或数据，不应该引起服务器上资源的状态变化。
    • POST请求：通常用于向服务器提交数据，如表单数据，并可能导致服务器上资源的状态变化或副作用，如创建、更新资源。

请求头

key value
Host 请求的主机名
User-Agent 浏览器版本，例如Chrome浏览器的标识类似Mozilla/5.0 … Chrome/79，IE浏览器的标识类似Mozilla/5.0 (Windows NT …) like Gecko
Accept 表示浏览器能接收的资源类型，如text/*，image/或者/*表示所有；
Accept-Language 表示浏览器偏好的语言，服务器可以据此返回不同语言的网页；
Accept-Encoding 表示浏览器可以支持的压缩类型，例如gzip, deflate等。
Content-Type 请求主体的数据类型。
Content-Length 请求主体的大小（单位：字节）。

相应协议

  • 与HTTP的请求一样，HTTP响应的数据也分为3部分：响应行、响应头 、响应体

响应状态码

  • 常见状态码：
    • 200 ok 客户端请求成功
    • 404 Not Found 请求资源不存在
    • 500 Internal Server Error 服务端发生不可预期的错误
状态码分类 说明
1xx 响应中 — 临时状态码。表示请求已经接受，告诉客户端应该继续请求或者如果已经完成则忽略
2xx 成功 — 表示请求已经被成功接收，处理已完成
3xx 重定向 — 重定向到其它地方，让客户端再发起一个请求以完成整个处理
4xx 客户端错误 — 处理发生错误，责任在客户端，如：客户端的请求一个不存在的资源，客户端未被授权，禁止访问等
5xx 服务器端错误 — 处理发生错误，责任在服务端，如：服务端抛出异常，路由出错，HTTP版本不支持等

状态码大全：https://cloud.tencent.com/developer/chapter/13553

响应头

响应头 说明
Content-Type 表示该响应内容的类型，例如text/html，application/json。
Content-Length 表示该响应内容的长度（字节数）。
Content-Encoding 表示该响应压缩算法，例如gzip。
Cache-Control 指示客户端应如何缓存，例如max-age=300表示可以最多缓存300秒。
Set-Cookie 告诉浏览器为当前页面所在的域设置cookie。

JavaWeb的概念

  • 指所有通过语言编写，可以通过浏览器访问的程序的总称。基于请求与响应来开发
  • 请求（Request）：客户端给服务器发送数据
  • 响应（Response）：服务器给客户端回传数据
  • 请求与响应是成对出现的

Web资源的分类

  • Web指所有浏览器可访问的程序，不单指Java
  • Web资源按实现的技术和呈现效果不同，又分为静态资源与动态资源
    • 静态资源：.html .css .js .txt .mp4 .jpg等
    • 动态资源：jsp页面，Servlet程序

常用的Web服务器

  • Tomcat：支持jsp与Servlet，轻量级，应用最广（7.*与8.*），免费
  • Jboss：免费，学习曲线高
  • GlassFish：商业级，强健
  • Resin：收费，应用较多
  • WebLogic：收费，应用广，适合大公司

Web服务器-Tomcat

  • Apache 软件基金会的开源免费的轻量级Web服务器。
  • 对HTTP协议的操作进行封装，使得程序员不必直接对协议进行操作，让Web开发更加便捷，主要功能是 “提供网上信息浏览服务”。
  • 支持servlet、jsp等少量javaEE规范。因此也被称为web容器、servlet容器。
  • 官网：https://tomcat.apache.org/

Tomcat目录

  • 目录下直接包含Tomcat的bin目录，conf目录等，我们称之为Tomcat的安装目录或根目录

  • bin：该目录下存放的是二进制可执行文件

  • conf：存放Tomcat服务器的配置文件，这个目录下有四个最为重要的文件：

    • server.xml：配置整个服务器信息。例如修改端口号。默认HTTP请求的端口号是：8080

    • tomcat-users.xml：存储tomcat用户的文件，保存tomcat的用户名及密码，以及用户的角色信息。

    • web.xml：部署描述符文件，这个文件中注册了很多MIME类型用来说明文档的内容是什么类型

    • context.xml：对所有应用的统一配置，通常我们不会去配置它。

  • lib：Tomcat的类库，里面存放服务器jar包，建议只把Tomcat需要的jar包放到这个目录下

  • logs：存放服务器运行时输出的日志文件

  • temp：存放Tomcat的临时数据，这个目录下的东西可以在停止Tomcat后删除

  • webapps：存放web项目的目录，其中每个文件夹都是一个项目，文件夹的名字就是项目名。

    ROOT项目是一个特殊的项目，在地址栏中访问：http://127.0.0.1:8080，没有给出项目目录时，对应的就是ROOT项目

    示例项目http://localhost:8080/examples

  • work：存放运行时生成的文件与最终运行的文件。存放服务器运行时jsp翻译为Servlet的源码与Session钝化（序列化，写到磁盘上）的目录

  • LICENSE：许可证。

  • NOTICE：说明文件。

Tomcat服务器操作

  • 启动：

    • 点击bin目录下的stratup.bat
    • 在命令行cd至bin目录，敲入启动命令catalina run

  • 测试启动成功：浏览器地址栏输入http://localhost:8080访问ROOT工程

  • 停止：

    • 点击服务器窗口关闭按钮
    • Ctrl+C
    • 点击bin目录shutdown.bat

  • 修改Tomcat端口号：conf内server.xml修改Connector标签下端口号（1000~65535），重启生效

  • 部署Web工程：将Web工程目录拷贝至webapps目录下

  • 访问Tomcat下的Web工程：

    • http://localhost:8080/工程名

    • conf目录下\Catalina\localhost\创建.xml配置文件

      写入：<Context path="/book" docBase="D:\book"/>

      path表示工程访问路径，docBase表示工程在本机上的目录（不一定要在webapps）

    • 找到工程下index.html拖入浏览器

手拖与输入地址访问区别

  • 手拖使用file协议，地址是file://项目本机地址，表示直接读取协议后路径解析并展示
  • 地址访问私有Http协议，地址是http://localhost:8080/工程名/资源名，表示客户端发送请求给服务器，服务器（Tomcat）读取资源文件回传，客户端再得到页面内容并解析展示

ROOT工程访问及默认index.html访问

  • 输入http://ip:port/无工程名时默认访问ROOT工程
  • 输入http://ip:port/工程名/项目名/无资源名，默认访问index.html页面

IDEA 中创建 Web 项目及部署 Tomcat

  1. 新建一个普通 Module

  2. 选择Add Framework Support，勾选Web Application

  3. 此时可以看见目录结构（其中index.jsp文件可以删除）

  4. 在Tomcat中部署 Web项目

    • 进入运行/调试配置==》编辑配置

    • 点击左上角的+，找到Tomcat Server，选择Local

    • 新建一个 Tomcat Server容器，进入Deployment选项卡部署 Web 项目。点击Artifact就可以选择想要部署的 web项目，建议部署后修改Name为对应项目名

    • 在最下面编辑该项目的应用目录Application context

      此处设置的值，会影响浏览器访问该项目的URL 的值。如果此处为/，则初始的 URL前缀就为http://localhost:8080/。假如此处为javaweb-demo，则 URL的前缀就为http://localhost:8080/javaweb-demo/

IDEA部署并运行项目的原理

  • idea并没有直接进将编译好的项目放入tomcat的webapps中
  • idea根据关联的tomcat,创建了一个tomcat副本,将项目部署到了这个副本中
  • idea的tomcat副本在C:\用户\当前用户\AppData\Local\JetBrains\IntelliJIdea2022.2\tomcat\中
  • idea的tomcat副本并不是一个完整的tomcat,副本里只是准备了和当前项目相关的配置文件而已
  • idea启动tomcat时,是让本地tomcat程序按照tomcat副本里的配置文件运行
  • idea的tomcat副本部署项目的模式是通过conf/Catalina/localhost/*.xml配置文件的形式实现项目部署的

WEB项目的标准结构

app 本应用根目录

  • src 存储Java源代码

  • static 非必要目录,约定俗成的名字,一般在此处放静态资源 ( css js img)

  • WEB-INF 必要目录,必须叫WEB-INF,受保护的资源目录,浏览器通过url不可以直接访问的目录

    • classes 必要目录,src下源代码,配置文件,编译后会在该目录下,web项目中如果没有源码,则该目录不会出现

    • lib 必要目录,项目依赖的jar编译后会出现在该目录下,若web项目没有依赖任何jar,则该目录不会出现

      引入jar包方法

      • 法一：将包拖入lib目录并右键选择Add as library
      • 法二：打开项目结构==》库 ==》左上角+号 ==》java ==》浏览选择到当前类库所需jar包 ==》选择使用模块

    • web.xml 必要文件,web项目的基本配置文件. 较新的版本中可以没有该文件,但是学习过程中还是需要该文件

  • index.html 非必要文件,index.html为默认的欢迎页
